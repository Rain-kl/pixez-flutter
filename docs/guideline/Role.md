# Role Guidelines

你现在要对一个 Flutter 项目进行二次开发。

我的核心要求是：
1. 尽量不要侵入式修改原项目代码。
2. 尽量减少未来同步上游代码时产生的 Git 冲突。
3. 我新增的功能、页面、组件、服务、状态管理、工具函数、资源等，尽量集中放在一个独立目录中。
4. 原项目代码只做最小必要接入，例如路由注册、菜单入口、依赖注入入口、配置开关等。
5. 所有新增代码要结构清晰、可维护、可删除、可迁移。
   请你在修改代码前，先完整分析项目结构，然后制定低侵入式二开方案。
   推荐约束如下：
- 新增功能统一放在：
  `lib/custom/`
  或者：
  `lib/extensions/`
  如果项目已有类似扩展目录，则优先复用项目现有风格。
- 新增代码不要散落在原项目各个目录中。
- 不要随意修改原有业务逻辑。
- 不要直接大改原有页面、组件、状态管理、网络请求层。
- 如果必须改原文件，请遵循“最小补丁原则”：
    - 只增加入口调用；
    - 不重构原逻辑；
    - 不移动原代码；
    - 不格式化整个文件；
    - 不做无关改动；
    - 不修改与当前需求无关的代码。
      建议目录结构：
```text
lib/custom/
  custom_entry.dart              # 自定义功能统一出口
  custom_routes.dart             # 自定义路由
  custom_menu.dart               # 自定义菜单/入口
  custom_config.dart             # 自定义配置
  pages/                         # 自定义页面
  widgets/                       # 自定义组件
  services/                      # 自定义服务
  models/                        # 自定义模型
  providers/ 或 controllers/      # 自定义状态管理
  utils/                         # 自定义工具函数
  constants/                     # 自定义常量
```

请优先采用以下方式接入原项目：

1. 路由扩展
- 如果项目使用 go_router，请新增 custom_routes.dart，在原路由文件中只添加一行合并自定义路由。
- 如果项目使用 Navigator 或命名路由，也要把自定义路由集中放在 custom_routes.dart。
- 不要直接把大量新路由写进原路由文件。

2. 菜单/入口扩展
- 如果需要在原 App 中增加入口，请创建 custom_menu.dart。
- 原菜单文件只添加一处合并逻辑，例如：

final menus = [
  ...originMenus,
  ...customMenus,
];

- 不要直接在原菜单结构中硬编码大量新增菜单项。

3. 配置开关
- 所有二开功能都要尽量支持开关控制。
- 新增 custom_config.dart，例如：

class CustomConfig {
  static const bool enableCustomFeature = true;
}

- 原项目接入时尽量通过开关判断，方便以后禁用自定义功能。

4. 状态管理
- 先识别项目使用的状态管理方案，例如 Provider、Riverpod、GetX、Bloc、MobX 等。
- 新增状态管理代码放在 lib/custom/providers/、lib/custom/controllers/ 或匹配项目风格的目录下。
- 不要随意改原有全局状态对象。
- 如果必须注册 provider/controller，只在原入口处添加最小注册逻辑。

5.网络请求
- 不要直接修改原有 API client。
- 自定义接口放在：

lib/custom/services/

- 如果必须复用原项目请求封装，可以在自定义 service 中调用原封装。
- 不要污染原 API 层。

6. 模型
- 自定义模型放在：

lib/custom/models/

- 不要修改原有 model，除非必须兼容原对象。
- 如果需要适配原数据结构，优先写 adapter，而不是改原 model。

7. UI 组件
- 自定义页面放在：

lib/custom/pages/

- 自定义组件放在：

lib/custom/widgets/

- 尽量复用原项目主题、颜色、字体、公共组件。
- 但不要修改原公共组件，除非只是增加极小的可选参数，并且保持完全向后兼容。

8. 资源文件
- 自定义资源集中放在：

assets/custom/

- 不要混入原有 assets 目录。
- 如果需要修改 pubspec.yaml，只新增自定义资源路径，不改动无关配置。

9. 依赖管理
- 不要随意添加第三方依赖。
- 如果确实需要新增依赖，必须说明：
    - 为什么需要；
    - 是否可以不用；
    - 是否会增加维护成本；
    - 是否会影响上游同步；
    - 是否有更轻量替代方案。
- 修改 pubspec.yaml 时只添加必要依赖，不格式化整个文件。

10. 代码风格
- 保持项目现有代码风格。
- 不要全局格式化。
- 不要重命名原有类、变量、文件。
- 不要做无关重构。
- 每次修改尽量小而清晰。

11. Git 冲突控制
- 所有修改都要按以下分类输出：
    - 新增文件；
    - 修改文件；
    - 修改原因；
    - 是否可能导致上游同步冲突；
    - 是否可以进一步降低侵入性。
- 对原项目文件的修改越少越好。
- 如果某个需求必须大改原项目，请先停下来说明原因，并给出更低侵入性的替代方案。

12. 交付要求
- 修改前先输出方案，不要直接大改。
- 方案中要明确：
    - 新增目录结构；
    - 需要接入哪些原文件；
    - 每个原文件为什么必须修改；
    - 是否有不修改原文件的替代方案；
    - 后续同步上游时如何处理。
- 修改后输出总结：
    - 新增了哪些文件；
    - 修改了哪些原文件；
    - 哪些地方是二开入口；
    - 如何关闭二开功能；
    - 如何整体删除自定义功能；
    - 上游同步时重点关注哪些文件。

请严格按照“低侵入、集中管理、可回滚、少冲突”的原则完成开发。
